### 11.2 风险管理预案表 (软工实验详尽版)

| 风险类别 | 具体风险描述 | 风险级别 | 应对预案 (Mitigation & Contingency Plan) |
| :--- | :--- | :--- | :--- |
| **技术风险** | **前端框架上手难**：第三版 Flutter 学习曲线陡峭，很多同学之前没接触过 Dart 语言，导致移动端开发卡壳，进度严重滞后。 | **高** | **预案**：确立 uni-app 作为保底方案（第二版）。设定“止损点”：若开发周期过半，前端同学仍无法跑通基础的“商品列表+详情页”数据渲染，果断放弃 Flutter，切回 uni-app 打包多端，确保结课时有实体产品可供演示。 |
| **技术风险** | **自动化部署(CI/CD)受阻**：在 GitHub Actions 上写部署脚本一直报错，受限于校园网环境或云服务器（频频失败。 | **中** | **预案**：前期预留 2 天专门尝试流水线；若一直失败，不要死磕，立即启动备用方案：编写一段简单的 Shell 脚本，甚至直接在本地用 Maven/npm 打包后，通过 Xftp/宝塔面板手动上传 `.jar` 或 `dist` 文件夹到服务器跑起来。 |
| **技术风险** | **Git 协作灾难（代码覆盖/冲突）**：团队成员不熟悉 Git 操作，导致 push 代码时产生大量冲突，甚至把别人写好的核心逻辑覆盖或弄丢了。 | **高** | **预案**：组长提前建好 doc分支，要求每人必须将文件传到对应的分支上。应急预案：如果发生代码丢失，立刻找组长通过 `git reflog` 或历史记录回滚；同时要求大家每天收工前把自己写的代码在本地打包一个 `.zip` 留作最原始的备份。 |
| **技术风险** | **“在我电脑上明明能跑” (环境不一致)**：后端的同学本地运行正常，但传到服务器上或者发给前端同学本地启动时，因为环境不一致，数据缺失等导致一直跑不起来。 | **中** | **预案**：后端必须在代码仓库里放一份标准的 和环境说明 `README.md`。如果环境实在调不通，直接将其部署到网页上，避免本地数据库配置问题。 |
| **需求风险** | **范围蔓延 (想得太多做不完)**：大家前期头脑风暴太嗨，想加入真实的支付接口、复杂的物流追踪、AI 智能推荐等超出课程要求的功能，导致最后连基础的购物车都没写完。 | **高** | **预案**：组长/PM 严格把控需求边界，坚持 MVP（最小可行性产品）原则。支付只做个“模拟成功”的弹窗，物流只做简单的文字状态变更，推荐列表直接按商品创建时间倒序排。在“登录-加购物车-下单”这个核心闭环没跑通前，绝对不允许写任何花里胡哨的额外功能。 |
| **需求风险** | **偏离实验评分标准**：做了一大堆酷炫的前端动画，但最后发现老师/助教的评分表里重点看的是“软件工程文档规范”或“数据库设计范式”，导致得分很低。 | **高** | **预案**：项目启动前，全组对照老师发的“实验指导书”和“评分细则”逐条勾选功能。对于占分比重大的模块（如测试报告、UML图），即使代码写得烂也要把文档补全；对于不计分的拓展功能，时间不够直接砍掉。 |
| **人员风险** | **进度冲突与请假**：临近期末，团队成员因为其他科目的课程设计、考研复习或生病请假，导致某一端（如前端或后端）进度完全停滞。 | **中** | **预案**：推行清晰的代码注释与文档制度。一旦有人实在没空，其他成员可以根据文档临时接手写简单的 CRUD 接口或画几个静态页面。同时，在任务分配上留出 20% 的缓冲时间。 |
| **人员风险** | **团队成员“划水” (消极怠工)**：个别同学分配了任务但迟迟不推进，发消息不回，严重拖累整体进度。 | **高** | **预案**：组长每周分配一定的任务，明确落实“这周谁、在什么时间前、交付什么文件”。遇到严重划水且沟通无效的，组内骨干成员需提前接管其核心任务，并在期末的“小组成员贡献度/互评表”中如实向老师反映，避免全组陪葬。 |
| **进度风险** | **前后端扯皮时间过长**：前端怪后端没给数据，后端怪前端传参不对，互相等，导致耗费大量时间。 | **高** | **预案**：“后端主程”必须先行出具一份简单的 API 接口文档（可以用 Apifox 或直接写 Markdown）。前端在后端真实接口没写完前，严格按照文档自己造 Mock 数据（假数据）开发。|
| **进度风险** | **验收演示（答辩）翻车**：在给老师演示项目时，网络断掉，或者服务器突然卡死等，导致程序点不开，场面尴尬。 | **高** | **预案**：这是学生项目最大的风险！**必须提前录制一份完整的、流畅的项目操作演示视频**（带讲解）。如果现场演示服务器崩了或网络连不上，立刻向老师说明情况，并播放备用视频证明项目确实做出来了。 |
