# 可行性分析(研究)报告(FAR)

## 1 引言

### 1.1 标识
* **项目名称**：商品网上交易系统
* **文档标识**：可行性分析报告（FAR）- 完整版

### 1.2 背景
本项目是在软件工程课程实践要求下提出的模拟商业项目。旨在开发一个能够支持用户在线注册、发布商品、审核商品以及完成在线交易全流程的 C2C/B2C 电商平台。我们的目标是通过团队协作，在有限周期内高效实现一套功能完整的标准 Web 与移动端业务系统，并引入现代化的企业级工程实践，包括CI/CD，多端适配等。

### 1.3 项目概述
本项目是一个跨平台的商品网上交易系统。主要特性是用户管理、商品展示与管理、后台审查机制、在线购物及订单配送状态追踪。项目的开发方为 4 人学生敏捷开发小组，目标用户为普通网络购物者及个人卖家。

### 1.4 文档概述
本文档旨在对“商品网上交易系统”进行初期可行性分析，从技术路线、CI/CD 工程化落地、开发模型选择、风险控制、经济和法律等方面综合论证项目的可行性，作为后续项目立项开发的决策依据。

## 3 可行性分析的前提

### 3.1 项目的要求
本项目必须实现以下五个模块：
1. **用户管理**：主要包括注册、登录、个人中心等功能模块。
2. **商品管理**：主要包括发布在线产品展示功能（支持图文）以及对商品信息的增删改查。
3. **审查管理**：主要包括管理员对新注册用户的审查以及对上架商品的合规性审查。
4. **用户购物**：用户能对已审核通过的商品进行检索、浏览，并完成在线下订单购物操作。
5. **用户体验**：用户可以收藏产品，同时会根据用户的收藏给用户推荐相似的产品
6. **AI辅助**：用户可以与AI对话，让其介绍产品并进行推荐
7. **配送与订单流转**：买卖双方对订单的状态操作（用户下单 -> 卖家出货 -> 买家收货确认 -> 交易闭环）。

### 3.2 项目的目标
在课程规定的时间内，采用敏捷迭代的开发模式，交付一个前后端分离、支持多端访问、具备自动化部署流水线且无明显重大 Bug 的在线交易系统。

### 3.3 项目的环境、条件、假定和限制
* **环境限制**：团队共 4 人（产品、前后端、测试），需在较短的周期内完成。
* **资金限制**：项目无实际商业预算，需全部采用开源免费的技术框架、云服务免费额度和本地服务器。

## 4 可选的方案与可行性技术路线

为了应对从 Web 端向移动端拓展的演进需求，团队摒弃了传统的一体化渲染方案，决定采用**“前后端分离 + 渐进式多端演进”**的可行性技术路线。迭代方案如下：

### 4.1 第一版（ MVP）：Django + Vue
* **技术架构**：后端采用 Python Django 提供 API；前端采用 Vue.js 框架构建 Web 单页应用。
* **可行性分析**：Django 具备较高开发效率；Vue 作为前端主流框架，组件化开发可提升页面交互体验。此版本旨在以最快速度完成项目基本需求的 Web 端闭环。

### 4.2 第二版（轻量级移动端）：Django + Vue + uni-app
* **技术架构**：后端 API 保持不变；在原有 Vue Web 端的基础上，引入 uni-app 框架开发移动端应用。
* **可行性分析**：uni-app 同样基于 Vue 语法，团队前端人员学习成本极低。通过一次编写代码，即可快速编译出微信小程序、H5 移动端网页甚至 App，以极低的成本极大拓宽了系统的移动端用户覆盖面。

### 4.3 第三版（高性能原生级）：Django + Vue + Flutter
* **技术架构**：复用后端 API ；保留 Vue 作为 Web 端；使用 Google 开源的 Flutter 框架重构移动端 App。
* **可行性分析**：uni-app 对动画的支持有限，且在原生移动端表现极差。Flutter 采用自绘引擎，能提供媲美原生 iOS/Android 的体验。

**最终路线选择**：
遵循增量开发原则，团队将严格按照上述一、二、三版的路线进行渐进式开发，确保每一个里程碑都有可交付、可运行的产品。

## 5 所建议的系统与 CI/CD 工程化实践

### 5.1 对所建议的系统的说明
系统整体采用 B/S 与 C/S 混合架构。后端统一由 Django 提供 API；前端根据不同的演进阶段，利用 Vue、uni-app 或 Flutter 进行多端消费，实现前后端解耦。

### 5.2 数据流程和处理流程
* **注册与发布**：用户注册 -> 触发数据校验 API -> 等待管理员后台审查 -> 审查通过 -> 用户通过前端组件上传图片及商品数据 -> 存入数据库待审 -> 审查后上架展示。
* **交易与配送**：买家多端搜索商品 -> 发起下单请求 -> 生成订单写入数据库 -> 卖家后台查收 -> 卖家调用 API 修改为“已出货” -> 买家确认收货 -> 交易闭环。

### 5.3 CI/CD 
为了保障上述多阶段技术路线的高效推进，降低 4 人团队的代码合并冲突与部署风险，本项目全面引入 CI/CD 自动化流水线，依托 GitHub Actions ：

1. **持续集成 (Continuous Integration - CI)**：
   * **代码提交与规范化检查**：任何团队成员向代码仓库 `push` 时，git hook自动触发代码风格检查。
   * **自动化测试**：自动化运行后端接口的单元测试 django test，如果用例失败，将阻止代码合并。
2. **持续交付/部署 (Continuous Deployment - CD)**：
   * **自动化构建**：代码合并至主分支后，CI/CD 工具会自动将Flutter代码打包，并构建为可安装安卓.apk, IOS .ipa软件包。
   * **自动化部署**：通过部署脚本，将后端服务器推送到测试服务器进行部署。前端静态资源自动更新到 Nginx 目录，后端 Django 自动执行数据库迁移并重启服务。

## 10 软件开发过程模型比较与项目应用

为保证 4 人团队在有限周期内交付，团队在立项初期对软件开发过程模型进行了充分讨论。

### 10.1 传统模型与敏捷开发的比较

| 比较维度 | 传统软件开发模型 | 敏捷开发 |
| :--- | :--- | :--- |
| **核心思想** | 阶段划分严格，线性推进（需求->设计->编码->测试）。 | 迭代式、增量式开发，拥抱需求变化。 |
| **优点** | 流程清晰，文档规范，阶段里程碑明确，适合需求稳定、大型且人员流动率高的传统外包项目。 | 交付速度快，用户反馈周期短；团队沟通紧密；能迅速适应课程作业中可能出现的需求微调或技术栈演进。 |
| **缺点** | 难以适应需求变更；测试介入晚，前期隐藏的风险到项目后期才暴露；文档工作量巨大。 | 对团队成员的自驱力要求高；若缺乏纪律容易变成“无计划的游击战”；对文档的重视程度相对较低。 |
| **主流方法** | 瀑布模型、V 模型、螺旋模型等。 | **Scrum**，固定周期的冲刺与角色划分；并使用**Kanban**，工作流可视化。 |

### 10.2 在本项目中的应用策略与团队分工

**应用策略：结合 Scrum 与看板的敏捷开发**
鉴于本项目的渐进式技术路线，团队一致认为传统瀑布模型完全不适用。我们将采用敏捷开发中的 **Scrum** 框架，辅以 **Kanban（看板）** 进行任务追踪，以 1-2 周为一个冲刺周期（Sprint）推进研发。

**小组 4 人角色、工程开发与规范文档分工：**
根据课程规范与实际开发需求，我们将团队划分为以下 4 个角色，并严格界定了各自的工程研发任务与标准化文档产出：

## 11 项目风险管理与应对预案

基于 4 人团队规模和较短的开发周期，我们在开发过程中可能面临技术、进度、需求等多维度的风险。团队对潜在风险按照**发生概率 × 影响程度**进行了分级，并制定了详细的应对预案。

### 11.1 风险分级标准
* **高风险**：发生概率较高，且一旦发生将直接导致项目延期或核心功能不可用。
* **中风险**：发生概率一般，会对项目进度或非核心功能产生影响。
* **低风险**：发生概率低，可通过短时间增加工作量解决。

### 11.2 风险管理预案表

| 风险类别 | 具体风险描述 | 风险级别 | 应对预案 (Mitigation & Contingency Plan) |
| :--- | :--- | :--- | :--- |
| **技术风险** | 第三版 Flutter 学习曲线陡峭，导致移动端开发卡壳。 | **高** | **预案**：将 uni-app 作为保底方案。若开发周期过半仍无法突破 Flutter 难点，放弃第三版，使用 uni-app 打包多端交付，保证有可用产品。 |
| **技术风险** | CI/CD 自动化部署脚本无法跑通。 | **中** | **预案**：前期进行流水线调试；若受限于服务器环境一直失败，编写 Shell 部署脚本，改用半自动化手动部署。 |
| **需求风险** | 范围蔓延：后期想加入真实的支付接口、复杂的物流追踪等超出课程要求的功能，导致做不完。 | **高** | **预案**：PM严格把控 Backlog 边界，坚持 MVP原则。在主要流程闭环前，拒绝新增额外需求。 |
| **人员风险** | 团队成员因学业繁重或其他事务请假，导致某端进度完全停滞。 | **中** | **预案**：通过统筹者推行严格的代码规范与 Git 提交制度。一旦出现人员缺席，其他成员可根据完善的文档临时接手紧急 Bug 修复等。 |
| **进度风险** | 前后端接口联调不一致，耗费大量时间扯皮。 | **高** | **预案**：主程必须先行出具标准 API 接口文档；前端/测试运维在后端真实接口未完成前，按照文档使用 Mock 数据开发。 |
