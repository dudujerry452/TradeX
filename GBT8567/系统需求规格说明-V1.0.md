# 系统需求规格说明

## 1 引言

### 1.1 标识

- **项目名称**：商品网上交易系统
- **文档标识**：系统需求规格说明-V1.0

### 1.2 系统概述

本系统为软件工程课程实践要求下开发的模拟商业电商平台，旨在实现用户在线注册、商品发布与审核、在线购物、订单配送全流程闭环的跨平台交易系统。

**系统用途**：为普通网络购物者、个人卖家提供标准化的线上商品交易服务，同时为平台管理员提供合规审查与系统管理能力。

**系统一般特性**：采用前后端分离架构，支持 Web 端、移动端（APP、小程序）多端访问，核心业务覆盖用户管理、商品管理、审查管理、评论管理、购物交易、订单流转六大模块。

**开发与运维历史**：本系统为全新开发项目，无历史开发与运维记录。

**项目相关方**：

- ***投资方***：无
- ***需方***：软件工程课程教学单位
- ***最终用户***：普通网络购物者、个人卖家、平台管理员
- ***开发方***：4人学生敏捷开发小组
- ***支持机构***：软件工程课程教学团队
- ***运行现场***：互联网公网环境，支持PC端浏览器、APP、小程序等多终端访问。

## 3 需求

### 3.1 要求的状态和方式

系统需支持多种运行状态，不同状态对应差异化的功能与权限需求，具体状态定义如下：

**未登录状态**：用户未完成身份认证，仅可访问系统首页、商品公开检索与详情页，不可使用下单、商品发布、添加到购物车、个人中心等需授权功能。

**普通买家登录状态**：用户完成注册审核并登录，可使用商品检索、购物车、下单、订单管理、收货确认、个人信息管理等买家全量功能。

**个人卖家登录状态**：用户完成卖家资质审核并登录，在买家功能基础上，额外具备商品发布、商品管理、店铺管理、订单发货、售后处理等卖家功能。

**平台管理员登录状态**：管理员账号登录，具备用户资质审核、商品合规性审查、用户权限管理、系统日志查看、数据统计、系统配置等全量管理功能。

**系统维护状态**：系统升级或故障处理时的运行状态，仅管理员可访问后台。

本规格说明中每个需求均与上述状态和方式关联，无关联的需求默认适用于所有正常运行状态。

### 3.2 需求概述

#### 3.2.1 系统总体功能和业务结构

本系统为跨平台商品交易系统，核心业务闭环为：用户注册资质审核→卖家商品发布合规审核→买家商品检索与下单→订单配送状态流转→买家确认收货完成。系统整体采用前后端分离的分层架构，分为四层结构：

**接入层**：面向多终端的前端适配层，支持Web端、移动端（APP、小程序）等多终端访问。

**业务层**：核心业务能力模块，包含用户管理、商品管理、审查管理、评论管理、购物交易、订单流转六大模块。

**数据层**: 负责系统数据的存储、读取与管理，包含用户数据、商品数据、订单数据、评论数据等（详见数据库（顶层）设计说明）。

**基础设施层**: 提供容器化部署、安全防护等基础支撑能力。

#### 3.2.2 硬件系统的需求

**服务器端硬件需求**：无特殊需求，基础服务器或个人电脑均可。

**开发端硬件需求**：开发人员需配备普通 PC（推荐至少 8GB 内存以运行前端编译及 Docker），CI/CD 自动化依赖于代码托管平台的云端服务器。

**用户端硬件需求**：无特殊硬件要求，主流 PC、智能手机、平板电脑均可正常访问与使用系统。

#### 3.2.3 软件系统的需求

**服务器端软件需求**：操作系统（Windows/Mac/Linux），后端（Python/Django），环境隔离（Docker），数据库（SQLite）。

**开发端软件需求**：Windows/Mac/Linux 操作系统、Node.js 16+、Git 2.30+、Vue 3+ 前端框架、uni-app/Flutter 跨端开发框架、代码编辑与测试工具。

**用户端软件需求**：PC端支持Chrome、Edge、Firefox等主流现代浏览器。移动端支持安卓端APP、微信最新版本（小程序场景）。

#### 3.2.4 接口需求

本条明确硬件系统与软件系统之间的接口需求：

1. 软件需适配用户端PC的键盘、鼠标、USB等输入输出接口，支持常规人机交互操作。
2. 软件需适配移动端智能手机的触摸屏、定位模块等硬件接口，支持商品图片上传、收货地址定位等功能。
3. 软件需适配终端设备的网络适配器接口，支持以太网、WiFi、5G/4G等网络通信方式，可正常完成数据传输与交互。

### 3.3 系统能力需求

本条分条详细描述系统每一核心能力对应的需求，每个需求分配唯一标识符，明确功能要求、性能指标与异常处理规则。

#### 3.3.1 用户管理能力（SR-3.3.1）

##### 3.3.1.1 用户注册能力（SR-3.3.1.1）

1. 系统需支持用户通过手机号或邮箱完成账号注册，注册信息需包含用户名、登录密码、联系方式，密码需满足复杂度要求。
2. 重复用户名、手机号、邮箱需实时给出明确的错误提示。
3. 用户注册完成后，账号自动进入待审核状态，不可直接使用需授权功能。

##### 3.3.1.2 用户登录与身份认证能力（SR-3.3.1.2）

1. 系统需支持账号密码登录等登录方式。
2. 需实现基于角色的权限控制，不同角色（买家、卖家、管理员）对应不同的接口访问权限，禁止越权调用。

##### 3.3.1.3 个人中心管理能力（SR-3.3.1.3）

1. 登录用户可查看、编辑个人基本信息，修改登录密码，绑定/换绑手机号/邮箱。
2. 支持用户管理收货地址，包含新增、编辑、删除、设置默认地址功能。
3. 支持用户查看/发布商品评价。
4. 支持用户收藏/取消收藏商品。
5. 支持用户关注/取消关注卖家。
6. 买家用户可查看个人全部订单记录、浏览历史、购物车数据、交易状态等信息。
7. 卖家用户可查看店铺信息、关注信息、商品发布记录、订单管理数据、交易统计信息。
8. 所有信息修改后需实时同步至数据库，保证多终端数据一致性。

#### 3.3.2 商品管理能力（SR-3.3.2）

##### 3.3.2.1 商品发布能力（SR-3.3.2.1）

1. 审核通过的卖家用户可发布商品，支持填写商品名称、分类、价格、库存、规格、详情描述等基础信息。
2. 支持商品图文上传，支持 JPG、PNG、WebP 主流格式。
3. 商品发布完成后商品自动进入待审核状态，不可直接上架展示。
4. 支持卖家对未审核的商品进行编辑、撤回操作，编辑后重新进入待审核队列。

##### 3.3.2.2 商品上下架与编辑能力（SR-3.3.2.2）

1. 审核通过的商品自动上架，可在前端页面展示与检索。
2. 卖家可对已上架商品修改库存、详情信息，修改后需重新进入审核流程，审核通过后更新展示内容。
3. 卖家可手动下架已上架商品，下架后商品不在前端展示，不可被下单购买。
4. 商品库存扣减需支持事务控制，防止超卖，库存为0时商品自动禁止下单。

##### 3.3.2.3 商品检索与展示能力（SR-3.3.2.3）

1. 系统需支持商品关键词模糊搜索，搜索结果支持按价格、销量、上架时间排序。
2. 支持按商品分类、价格区间、库存状态进行筛选，筛选结果实时刷新。
3. 商品详情页需完整展示商品图片、价格、规格、库存、卖家信息、详情描述。
4. 未登录用户可正常访问商品检索与详情页，无访问限制。

##### 3.3.2.4 商品推荐能力 （SR-3.3.2.4）

1. 系统需支持在主页展示热门商品、最新商品等。
2. 系统需支持调用deepseek等AI模型，实现用户与模型的对话交互，并根据用户需求推荐商品。

#### 3.3.3 审查管理能力（SR-3.3.3）

##### 3.3.3.1 用户资质审查能力（SR-3.3.3.1）

1. 管理员可在后台查看待审核用户列表，展示用户注册信息、提交时间。
2. 支持管理员对用户注册申请执行通过/驳回操作，系统实时推送至用户端。

##### 3.3.3.2 商品合规性审查能力（SR-3.3.3.2）

1. 管理员可在后台查看待审核商品列表，展示商品详情、卖家信息、提交时间。
2. 支持管理员对商品上架申请执行通过 / 驳回操作，系统实时推送至卖家端。
3. 支持管理员对已上架违规商品执行强制下架操作。

#### 3.3.4 购物交易能力（SR-3.3.4）

##### 3.3.4.1 购物车管理能力（SR-3.3.4.1）

1. 登录买家用户可将商品加入购物车，支持修改商品数量、删除商品、选择/取消选择结算商品。
2. 购物车数据需实时同步至服务端，保证多终端数据一致性。
3. 商品下架、库存不足时，需在购物车中给出明确提示，禁止该商品被结算。

##### 3.3.4.2 订单生成与下单能力（SR-3.3.4.2）

1. 系统需支持用户从购物车批量结算或商品详情页单独下单，生成订单前需校验商品库存、状态、价格有效性。
2. 生成订单时需同步扣减对应商品库存，采用事务控制保证数据一致性，防止超卖。
3. 订单需包含唯一订单编号、商品信息、价格、收货地址、下单时间、订单状态等核心信息。
4. 库存不足、商品下架、价格变动等异常情况，需立即终止下单流程，给出明确的错误提示，禁止生成异常订单。

#### 3.3.5 订单与配送管理能力（SR-3.3.5）

##### 3.3.5.1 订单状态管理能力（SR-3.3.5.1）

1. 系统需支持订单全生命周期状态管理，核心状态包括：待付款、待发货、待收货、已完成、已取消。
2. 卖家可对 “待发货” 状态的订单执行发货操作，填写物流公司、物流单号，操作后订单状态自动更新为 “待收货”。
3. 买家可对 “待收货” 状态的订单执行确认收货操作，操作后订单状态自动更新为 “已完成”，交易闭环完成。
4. 支持订单超时自动处理：待付款订单超时 24 小时未支付自动取消，恢复商品库存。待收货订单超时 15 天未确认自动完成，交易闭环。
5. 支持买家对已完成订单进行评价，审核通过后发布。

##### 3.3.5.2 配送信息管理能力（SR-3.3.5.2）

1. 卖家发货后，系统需实时同步物流信息至买家端，买家可查看物流单号、物流公司、物流轨迹。
2. 买卖双方均可在订单详情页查看完整的订单状态流转记录，包含操作人、操作时间、操作内容，不可篡改。

##### 3.3.5.3 售后管理能力（SR-3.3.5.3）

1. 买家可对 “已完成” 状态的订单执行售后操作（退货、换货），填写售后原因，操作后订单状态自动更新为 “售后中”。
2. 卖家可对售后操作进行评价，审核通过后发布。
3. 支持买家对售后操作进行取消操作，售后状态自动更新为 “售后取消”。
4. 审核后的换货订单可生成新的订单编号，与原订单编号区分。

#### 3.3.6 系统管理能力（SR-3.3.6）

1. 管理员可在后台进行用户权限管理。
2. 支持管理员查看系统核心数据统计，包括用户注册量、商品发布量、订单交易量、审核通过率等，支持数据导出。
3. 支持管理员配置系统基础参数，包括密码复杂度规则、商品审核规则等，配置修改实时生效，无需重启系统。

#### 3.3.7 用户交流评论能力（SR-3.3.7）

1. 登录买家可对已完成订单进行评价，审核通过后发布。
2. 支持买家对卖家评价进行点赞，评价内容实时刷新。
3. 支持买家与卖家之间的聊天，聊天内容实时刷新。

### 3.4 系统外部接口需求

本条描述系统与外部实体的接口需求，每个接口分配唯一标识符，明确接口约束与特性。

#### 3.4.1 接口标识和接口图

| 接口唯一标识符    | 接口名称         | 接口实体              | 接口类型    |
| ---------- | ------------ | ----------------- | ------- |
| IF-EXT-001 | 用户端人机交互接口    | 系统与终端用户（PC、手机、平板） | 人机交互接口  |
| IF-EXT-002 | 前后端 API 交互接口 | 系统后端服务与前端多终端应用    | 应用程序接口  |
| IF-EXT-003 | AI服务接口       | 系统与第三方AI服务商       | 第三方服务接口 |
| IF-EXT-004 | 数据库访问接口      | 系统业务服务与数据库        | 数据访问接口  |
| IF-EXT-005 | 网络通信接口       | 系统与互联网公网          | 通信协议接口  |

所有接口均采用标准化设计，接口实体的固定特性已明确，开发过程中需对系统侧施加对应的接口需求，保证接口兼容性与稳定性。

#### 3.4.2 用户端人机交互接口（IF-EXT-001）

1. 接口优先级：P0（核心）
2. 接口类型：人机交互接口，用于实时人机交互，是数据输入输出接口。
3. 数据元素特性：支持用户输入文本、图片、鼠标动作，系统输出文本、图片、音视频、交互控件，输入输出数据需符合系统业务规则，敏感数据需脱敏展示（如密码）。
4. 视听特性：界面布局需适配终端屏幕尺寸，字体、图标、颜色符合主流 UI 设计规范。

### 3.5 系统内部接口需求 （SR-3.5）

系统内部接口为后端各业务模块之间、前端各组件之间的交互接口，需求如下：

1. 内部接口需具备幂等性，所有写操作接口必须支持事务控制，保证数据一致性。
2. 前端各组件之间的接口，采用Vue组件化通信规范，保证组件间低耦合、高内聚。
3. 所有内部接口必须编写完整的接口文档，明确请求参数、响应格式、异常处理规则。

### 3.6 系统内部数据需求 （SR-3.6）

本条明确系统内部数据库、数据文件的核心需求：

1. 系统核心数据分为用户数据、商品数据、订单交易数据、用户评价及聊天相关数据等数据。
2. 用户敏感数据（登录密码、手机号、邮箱、收货地址）需加密存储，禁止明文存储。
3. 数据库设计需遵循第三范式，减少数据冗余，保证数据一致性，核心业务表必须创建合适的索引，提升查询效率。
4. 订单交易数据等核心数据不可篡改，保证数据可追溯。

### 3.7 适应性需求 （SR-3.7）

本条明确系统的适应性需求：

1. 系统前端页面需支持响应式布局，根据用户终端屏幕尺寸、分辨率自适应调整页面布局，适配PC端、移动端、平板等多终端设备。
2. 系统需支持运行环境的灵活适配，后端服务可部署在Linux等主流操作系统，支持物理机、虚拟机、Docker容器多种部署方式。
3. 系统需支持业务规模已经核心功能的扩展，无需重构核心架构。

### 3.8 安全性需求 （SR-3.8）

本条描述系统防止安全风险、降低安全隐患的核心需求：

1. 系统需具备应用安全防护能力，可有效防御 SQL 注入等常见Web攻击。
2. 系统需实现严格的权限控制，基于角色的访问控制模型，禁止越权访问、越权操作，普通用户不可访问管理员后台，卖家不可修改其他卖家的商品与订单数据。
3. 系统需具备数据安全防护能力，用户敏感数据加密存储，数据传输全程加密，防止数据泄露、篡改、窃取。
4. 系统需具备业务安全防护能力，订单生成采用事务控制，防止超卖、错单，接口具备幂等性，防止重复下单、重复发货。
5. 系统需具备账号安全防护能力，密码复杂度强制校验，多次登录失败自动锁定，支持异地登录提醒，防止账号被盗。

### 3.9 保密性和私密性需求 （SR-3.9）

本条明确系统的保密性和私密性需求：

1. 系统需严格遵循用户隐私保护相关法律法规，收集用户信息仅用于系统核心业务流程，不得超范围收集、使用、泄露用户个人信息。
2. 用户个人隐私信息（手机号、邮箱、收货地址、交易记录）仅授权相关人员可查看，前台页面需脱敏展示，禁止完整明文展示。
3. 系统需支持用户注销账号，账号注销后，按法律法规要求清除相关个人信息。
4. 系统不得将用户个人信息、交易数据提供给第三方，法律法规另有规定的除外。
5. 系统开发、运维人员需遵守保密协议，禁止私自查看、导出、泄露用户隐私数据。

### 3.10 操作需求 （SR-3.10）

本条明确系统的操作需求：

#### 3.10.1 常规操作需求 （SR-3.10.1）

本条明确系统的可使用性、可维护性、可移植性、可靠性和安全性需求：

3. 系统需提供完善的操作提示、异常提示、帮助文档，用户可快速解决使用过程中的常见问题。

#### 3.11.2 可维护性需求 （SR-3.11.2）

1. 系统采用模块化、前后端分离架构，模块间低耦合、高内聚，单个功能模块的修改不影响其他模块的正常运行。

#### 3.11.3 可移植性需求 （SR-3.11.3）

1. 系统后端服务支持部署在Linux等主流操作系统，支持物理机、虚拟机、Docker 容器多种部署方式，可快速迁移至不同的云服务器、运行环境。

4. 系统不依赖特定硬件设备，可在主流架构服务器上正常运行，可移植性强。

#### 3.11.4 可靠性需求 （SR-3.11.4）

1. 系统核心接口请求成功率≥99.9%，订单数据、交易数据准确率 100%，无数据丢失、错单、超卖等问题。
2. 系统支持高并发访问，单服务器可支持≥100并发用户同时访问，核心业务无异常。
3. 系统具备异常容错能力，单个非核心模块故障不影响核心交易业务的正常运行，数据库、服务节点故障可快速恢复，保证业务连续性。

#### 3.11.5 安全性需求 （SR-3.11.5）

与本规格说明3.8安全性需求、3.9保密性和私密性需求保持一致，所有安全需求必须100%实现，无安全漏洞。

### 3.12 故障处理需求 （SR-3.12）

本条明确系统的故障处理需求：

2. 数据库操作失败时，需立即执行事务回滚，保证数据一致性，防止出现脏数据、数据不一致问题。
3. 系统服务崩溃、进程异常退出时，需及时重启服务，重启后自动恢复业务，记录异常日志。
   前端页面加载失败、接口请求超时，需给出明确的提示，支持用户刷新重试，无页面白屏、卡死问题。
4. 系统出现业务异常（如库存不足、订单状态异常）时，需立即终止业务流程，给出明确的错误提示，禁止生成异常数据。

## 8 注解

### 8.1 背景信息

本系统为软件工程课程实践项目，旨在通过团队协作开发一套完整的跨平台商品网上交易系统，掌握企业级软件工程开发流程、前后端分离架构、CI/CD工程化实践、多端适配开发等核心技能。

### 8.2 术语定义

1. **C2C**：消费者与消费者之间的电子商务模式，个人卖家与个人买家通过平台完成交易。
2. **B2C**：企业与消费者之间的电子商务模式，企业通过平台向个人消费者销售商品。
3. **前后端分离**：一种软件架构模式，前端负责页面渲染与用户交互，后端负责业务逻辑与数据处理，通过API接口实现数据交互，前后端解耦。
4. **RESTful API**：一种API设计规范，基于HTTP协议，采用统一的请求方式、命名规范，实现标准化的接口设计。
5. **CI/CD**：持续集成与持续交付/部署，通过自动化流水线实现代码自动检查、测试、构建、部署，提升开发效率与交付质量。
6. **MVP**：最小可行产品，指具备核心业务功能、可交付运行的最小产品版本。
7. **SPA**：单页应用，指只有一个Web页面的应用，通过动态重写页面内容实现交互，无需页面刷新，提升用户体验。
8. **Docker**：一种开源的容器化引擎，可将应用与依赖打包到标准化容器中，实现跨环境的一致运行。
9. **MTBF**：平均无故障时间，指系统两次故障之间的平均运行时间，是衡量系统可靠性的核心指标。
10. **RBAC**：基于角色的访问控制，一种权限管理模型，通过角色分配权限，实现用户与权限的解耦。
