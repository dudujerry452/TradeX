# 商品网上交易系统数据库（顶层）设计说明（DBDD）

## 1 引言

### 1.1 标识

- **项目名称**：商品网上交易系统
- **文档标识**：数据库（顶层）设计说明-V1.0

### 1.2 数据库概述

本项目是一个基于 Web 的商品网上交易系统，是面向普通网络购物者及个人卖家的C2C/B2C电商平台 。主要特性涵盖用户管理、商品展示与管理、后台审查机制、在线购物及订单配送状态追踪 。项目的开发方为 4 人学生敏捷开发小组，目标用户为普通网络购物者及个人卖家 。本项目基于Python Django框架开发，采用B/S架构，核心功能涵盖用户管理、商品展示与管理、后台审查机制、在线购物及订单配送状态追踪。

本数据库设计为系统提供数据存储与管理支撑，确保业务逻辑闭环、数据一致性与完整性。确保满足系统五大核心模块（用户管理、商品管理、审查管理、用户购物、配送）的数据存储需求以及数据查询需求，支撑业务流程闭环。

### 1.3 文档概述

本文档为数据库顶层设计说明，明确数据库设计的目标、原则、整体架构、核心实体及关系，界定数据存储与访问规则，为后续详细数据库设计、开发及维护提供顶层指导。

## 3 数据库级设计决策

### 3.1 数据库输入与输出设计决策

本数据库的输入与输出完全匹配系统业务场景，核心设计如下：
**输入设计**：包括用户注册与身份信息、商品发布与更新信息、商品库存查询信息、注册审查操作信息、商品审查操作信息、订单创建与状态更新信息、商品留言与回复信息、管理员后台数据操作指令；所有输入均通过业务层参数校验后写入数据库，拒绝非法格式、越权范围的输入。
**输出设计**：产生的核心输出包括用户信息查询结果、商品列表与详情查询结果、审查记录与结果、订单全链路信息、商品留言内容、统计聚合数据；输出面向两类主体，对开发人员输出结构化数据对象，对终端用户输出脱敏后的业务展示数据，敏感字段默认不对外输出。
**接口设计**：数据库与业务系统的接口通过 Django ORM 框架实现，所有输入输出均封装为标准化的对象操作接口，无直接原生 SQL 暴露，接口设计与系统整体接口设计说明保持一致。

### 3.2 数据库行为设计决策

**核心业务响应逻辑**：用户注册提交后，自动写入用户表并标记注册状态为待审核；商品发布提交后，自动写入商品表并标记商品状态为待审核；订单创建时，同步校验商品库存、生成订单主记录与明细记录、扣减对应商品库存，全流程在原子事务内完成；订单状态变更时，同步更新对应时间节点字段。
**性能特性**：适配学生项目测试规模的访问压力，单条主键查询响应时间≤200ms，高频业务查询响应时间≤500ms，事务类操作执行时间≤1s；理论支持高并发商品查询场景。
**异常处理**：对不合法输入、越权操作、约束冲突的请求，直接拒绝执行并回滚事务，同时抛出可追溯的业务异常；对数据库运行异常，自动记录错误日志，保障业务数据不被污染。

### 3.3 用户呈现设计决策

本数据库主要面向后端程序员，不直接面向终端用户。数据库呈现设计主要考虑高效性、结构化、可维护性。
**高效性**：数据库设计基于关系型数据库模型，核心实体通过外键关联，保障数据参照完整性，理论支持高并发商品查询场景。
**结构化**：数据库表结构清晰、逻辑一致，符合关系型数据库设计规范，方便后续数据维护与扩展。
**可维护性**：数据库设计考虑了业务场景的扩展性，预留了必要的字段与索引，方便后续功能模块的添加与修改。

### 3.4 数据库管理系统选择设计决策

**数据库管理系统**：本数据库采用Django框架默认的SQLite数据库，轻量化、无需额外部署，适配Django框架默认配置，提升开发效率，其核心决策依据如下：
SQLite为开源免费产品，完全适配项目零商业预算的约束；
与最终选定的 Django 开发框架原生完美兼容，无需额外适配开发，大幅降低学习成本与开发周期；
支持完整的 ACID 事务特性，可保障订单交易场景的数据一致性；
社区生态完善，学习资料丰富，适配学生团队的开发能力；
预留了字段扩展、索引优化的空间，可适配需求的小幅变更。

### 3.5 安全性与保密性设计决策

**数据加密**：对敏感字段（如用户密码、支付信息）进行加密存储，防止数据泄露。具体来说，用户登录密码采用Django原生PBKDF2不可逆哈希算法加密存储，禁止明文存储；身份证号、手机号等隐私信息存储时做脱敏处理，展示场景仅暴露非完整信息。
**访问控制**：数据库账号按角色分配最小权限，业务运行账号仅具备基础增删改查权限，无库表结构修改、删库等高危操作权限；系统管理员与普通用户的业务数据访问权限完全隔离。
**注入攻击防护**：所有数据库操作均通过 ORM 预编译语句实现，禁止直接拼接SQL语句，从根源防范SQL注入攻击。
**数据备份**：定期备份数据库，保障数据持久化存储。

### 3.6 备份与恢复设计决策

**备份策略**：本地开发环境采用每日全量备份；测试环境采用每日增量备份 + 每周全量备份；备份数据保留周期不少于30天；
**备份约束**：备份操作以只读权限执行，不影响正常业务读操作，仅限制大批量写入操作；备份文件采用加密压缩存储，仅授权人员可访问；
**恢复策略**：支持全量数据恢复、单表数据恢复、指定行数据恢复；恢复操作执行前必须先对当前数据库做全量备份，避免恢复异常导致数据丢失。

### 3.7 索引与性能优化设计决策

所有数据表主键默认建立聚集索引，所有外键字段默认建立非聚集索引；
针对高频查询字段（如用户名、商品名称、订单状态、商品分类、注册审核状态、商品审核状态）建立适配的单值索引或联合索引；
避免过度索引，仅针对业务高频查询场景创建索引，保障写入性能不受影响；
支持自动空间回收，定期优化表碎片，保障数据库长期运行的性能稳定性。

## 4 数据库详细设计

### 4.1 数据库逻辑结构概述

本数据库采用关系型模型设计，共包含7个核心数据表，覆盖用户管理、商品管理、注册审查、商品审查、订单管理、订单明细、商品留言 7 大业务域，实体间通过主键 - 外键建立关联关系，完整支撑系统全业务流程闭环。

### 4.2 通用设计约定

1. 主键命名规则：`表名缩写_id`，统一为string类型，作为记录的全局唯一标识；
2. 外键命名规则：`关联表名缩写_id`，与关联表主键数据类型完全一致；
3. 枚举字段：统一采用string类型存储枚举值，明确限定枚举范围，保障数据一致性；
4. 时间字段：统一采用datetime类型，存储对应业务操作的完整时间戳；
5. 金额字段：统一采用decimal类型，避免浮点运算导致的精度丢失；
6. 敏感字段：加密存储或严格控制访问权限，禁止明文存储、全量对外展示。

## 5 用于数据库访问或操纵的软件配置项的详细设计

本章描述用于本数据库访问、操纵的全部软件配置项，所有配置项均基于Python Django Web框架实现，与系统整体技术栈完全兼容。

### 5.1 Django ORM框架（项目唯一标识符：DJANGO\_ORM）

1. 配置项概述：本配置项是系统核心的数据库访问层，实现Python业务对象与关系型数据库表的双向映射，封装所有数据库增删改查、事务管理、连接池管理能力，是业务系统与数据库之间的唯一交互通道，适配MySQL 8.0与SQLite 3双环境。
2. 设计决策：采用Django原生ORM框架，无需引入第三方组件，自带SQL注入防护、事务原子性保障能力，大幅降低开发成本，完全适配学生团队的开发能力与项目周期约束。
3. 约束与非常规特征：所有数据库操作必须通过ORM模型类实现，仅极端复杂查询场景可使用ORM原生查询接口，禁止直接拼接SQL语句；多表关联的业务操作必须包裹在ORM事务中执行，保障数据一致性。
4. 编程语言：Python 3.x，与系统CSCI指定开发语言完全一致。
5. 输入输出特性：输入为业务层传递的模型对象参数、查询过滤条件、事务执行指令；输出为结构化的ORM模型实例、查询结果集，与数据库表字段一一映射。
6. 接口特性：提供标准化的模型类定义接口、增删改查API、事务管理API、聚合查询API、关联查询API，与Django官方原生接口完全兼容。
7. 核心逻辑设计：
   - 系统启动时，自动完成ORM模型类与数据库表结构的映射校验，支持自动生成、执行表结构迁移脚本；
   - 接收业务层调用后，自动将对象操作转换为对应DBMS的标准化SQL语句，完成数据库执行；
   - 事务执行异常时，自动触发全量回滚，保障业务数据不被污染，并抛出可追溯的业务异常；
   - 查询执行完成后，自动将结果集映射为Python对象，返回给业务层处理。

### 5.2 Django Admin后台管理模块（项目唯一标识符：DJANGO\_ADMIN）

1. 配置项概述：本配置项是系统管理员用于数据库操纵的可视化管理模块，基于Django原生Admin模块定制开发，实现用户注册审查、商品审查、全量数据管理等后台业务的数据库操作。
2. 设计决策：复用Django原生Admin模块的权限控制、表单管理、列表展示能力，无需从零开发后台管理系统，完美适配项目审查管理核心需求，大幅缩短开发周期。
3. 约束与非常规特征：仅系统管理员角色可访问本模块，基于用户角色做细粒度权限控制；所有操作均自动记录操作日志，支持全流程追溯；禁止通过本模块直接修改订单金额、用户加密密码等核心敏感字段。
4. 输入输出特性：输入为管理员在可视化界面的操作指令、表单提交数据、审查结果与意见；输出为数据库对应表的增删改查结果、用户/商品状态更新、审查记录写入、操作日志生成。
5. 接口特性：提供模型注册接口、表单定制接口、权限控制接口、列表页定制接口、操作钩子接口，与Django Admin官方原生接口完全兼容。
6. 核心逻辑设计：
   - 管理员完成身份认证与权限校验后，基于角色权限展示可操作的数据模型与功能；
   - 管理员提交注册/商品审查操作后，自动更新对应用户/商品的状态字段，同步写入审查记录表，记录审核人、审核时间、审核意见；
   - 所有操作执行完成后，自动记录操作人、操作时间、操作内容、操作前数据快照，保障可追溯性。

### 5.3 业务系统功能模块（项目唯一标识符：BUSINESS\_MODULE）

1. 配置项概述：本配置项包含用户管理、商品管理、订单管理、留言管理等前端业务对应的数据库访问逻辑，按业务域拆分独立模块，是终端用户操作对应的数据库操纵单元。
2. 设计决策：按业务域拆分模块，每个业务模块对应独立的数据库访问逻辑，高内聚低耦合，便于维护、测试与迭代，完全匹配系统五大核心业务模块的需求。
3. 约束与非常规特征：所有用户操作必须先完成参数合法性校验与用户权限校验，仅允许操作当前用户有权限的数据；订单交易、库存扣减等核心操作必须在事务内执行；敏感操作需做二次校验，防止误操作。
4. 编程语言：Python 3.x，与系统CSCI指定开发语言完全一致。
5. 输入输出特性：输入为前端用户提交的表单数据、业务操作请求、查询过滤条件；输出为对应数据库表的更新结果、业务数据查询结果，经脱敏处理后返回给前端页面渲染。
6. 接口特性：按业务模块封装独立的Service层接口，向上为视图层提供标准化的数据库操作能力，向下调用DJANGO\_ORM配置项完成数据库访问，接口与业务场景一一对应。
7. 核心逻辑设计：
   - 接收前端请求后，先完成参数合法性校验、用户身份认证与权限校验，校验不通过直接拒绝执行；
   - 校验通过后，调用DJANGO\_ORM配置项执行对应的数据库操作，交易类操作自动开启事务；
   - 操作全部执行成功则提交事务，任一环节失败则触发全量回滚，返回对应异常信息；
   - 操作完成后，将执行结果经脱敏处理后返回给前端视图层。

### 5.4 数据库备份与恢复脚本（项目唯一标识符：BACKUP\_SCRIPT）

1. 配置项概述：本配置项是用于数据库定期备份、应急恢复的自动化Python脚本，保障数据库数据的安全性、完整性与可恢复性。
2. 设计决策：采用Python编写自动化脚本，结合系统定时任务执行，适配SQLite的备份恢复逻辑，无需额外引入商业备份工具，适配项目零预算约束。
3. 约束与非常规特征：备份脚本以只读权限访问数据库，避免影响正常业务运行；备份文件采用AES加密压缩存储，禁止明文存放；恢复操作必须由授权人员执行，执行前必须先备份当前数据库全量数据。
4. 过程式命令说明：脚本包含数据库全量备份命令、增量备份命令、备份文件加密命令、数据恢复命令、历史备份清理命令，所有命令均遵循SQLite官方规范，对应执行流程引用数据库官方用户手册。
5. 输入输出特性：输入为备份/恢复执行指令、备份周期配置、备份文件存储路径、加密密钥；输出为加密的数据库备份文件、备份/恢复操作执行日志、操作结果通知、过期备份文件清理。
6. 核心逻辑设计：
   - 定时任务触发后，脚本自动连接数据库，执行对应全量/增量备份操作；
   - 备份完成后，对备份文件进行加密压缩，存储到指定安全路径；
   - 自动扫描备份目录，清理超过保留周期的历史备份文件，释放存储空间；
   - 执行恢复操作时，先校验操作人权限，再对当前数据库做全量备份，之后执行恢复操作，完成后校验数据恢复完整性，返回执行结果。
