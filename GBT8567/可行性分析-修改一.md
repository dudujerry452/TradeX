* **项目名称**：商品网上交易系统
* **文档标识**：可行性分析报告（FAR）- 完整版

### 1.2 背景
本项目是在软件工程课程实践要求下提出的模拟商业项目。旨在开发一个能够支持用户在线注册、发布商品、审核商品以及完成在线交易全流程的 C2C/B2C 电商平台。提出者的核心要求和目标是通过团队协作，在有限周期内高效实现一套功能完整的标准 Web 与移动端业务系统，并引入现代化的企业级工程实践（如持续集成与多端适配）。

### 1.3 项目概述
本项目是一个跨平台的商品网上交易系统。主要特性涵盖用户管理、商品展示与管理、后台审查机制、在线购物及订单配送状态追踪。项目的开发方为 4 人学生敏捷开发小组，目标用户为普通网络购物者及个人卖家。

本文档旨在对“商品网上交易系统”进行初期的可行性分析，从技术路线、CI/CD 工程化落地、经济和法律等方面综合论证项目的可行性，作为后续项目立项和系统迭代开发的决策依据。

## 3 可行性分析的前提

### 3.1 项目的要求
本项目必须实现以下五大核心模块：
1. **用户管理**：主要包括注册、登录、个人中心等功能模块。
2. **商品管理**：主要包括发布在线产品展示功能（支持图文）以及对商品信息的增删改查。
3. **审查管理**：主要包括管理员对新注册用户的审查以及对上架商品的合规性审查。
4. **用户购物**：用户能对已审核通过的商品进行检索、浏览，并完成在线下订单购物操作。
5. **配送与订单流转**：买卖双方对订单的状态操作（用户下单 -> 卖家出货 -> 买家收货确认 -> 交易闭环）。

### 3.2 项目的目标
在课程规定的时间内，采用敏捷迭代的开发模式，交付一个前后端分离、支持多端访问、具备自动化部署流水线且无明显重大 Bug 的在线交易系统。

### 3.3 项目的环境、条件、假定和限制
* **环境限制**：团队共 4 人（兼顾产品、前后端、测试），需在较短的周期内完成。
* **资金限制**：项目无实际商业预算，需全部采用开源免费的技术框架、云服务免费额度或本地服务器。

## 4 可选的方案与可行性技术路线

为了应对从 Web 端向移动端拓展的演进需求，团队摒弃了传统的一体化渲染方案，决定采用**“前后端分离 + 渐进式多端演进”**的可行性技术路线。具体迭代方案如下：

### 4.1 第一版（核心 MVP）：Django + Vue
* **技术架构**：后端采用 Python Django (结合 Django REST Framework) 提供纯粹的 RESTful API；前端采用 Vue.js 框架构建 Web 单页应用（SPA）。
* **可行性分析**：Django 具备极快的开发效率和完善的 Admin 后台，能迅速满足项目的“审查管理”需求。Vue 作为前端主流框架，组件化开发能有效提升页面交互体验。此版本旨在以最快速度完成项目基本需求的 Web 端闭环。

### 4.2 第二版（轻量级移动端扩展）：Django + Vue + uni-app
* **技术架构**：后端 API 保持不变；在原有 Vue Web 端的基础上，引入 uni-app 框架开发移动端应用。
* **可行性分析**：uni-app 同样基于 Vue 语法，团队前端人员学习成本极低。通过一次编写代码，即可快速编译出微信小程序、H5 移动端网页甚至 App，以极低的成本极大拓宽了系统的移动端用户覆盖面。

### 4.3 第三版（高性能原生级演进）：Django + Vue + Flutter
* **技术架构**：后端 API 依旧稳定复用；保留 Vue 作为后台管理与 Web 端；使用 Google 开源的 Flutter 框架重构移动端 App。
* **可行性分析**：随着业务复杂度和用户对体验要求的提升，uni-app 在复杂动画和性能上可能存在瓶颈。Flutter 采用自绘引擎，能提供媲美原生 iOS/Android 的丝滑体验。作为架构的最终形态，此方案展现了系统极强的可扩展性。

**最终路线选择准则**：
遵循敏捷开发原则，团队将严格按照上述一、二、三版的路线进行渐进式开发，确保每一个里程碑都有可交付、可运行的产品。

## 5 所建议的系统与 CI/CD 工程化实践

### 5.1 对所建议的系统的说明
系统整体采用 B/S 与 C/S 混合架构。后端统一由 Django 提供高内聚的微服务化 API；前端根据不同的演进阶段，利用 Vue、uni-app 或 Flutter 进行多端消费，实现彻底的前后端解耦。

### 5.2 数据流程和处理流程
* **注册与发布**：用户注册 -> 触发数据校验 API -> 等待管理员后台审查 -> 审查通过 -> 用户通过前端组件上传图片及商品数据 -> 存入数据库待审 -> 审查后上架展示。
* **交易与配送**：买家多端搜索商品 -> 发起下单请求 -> 生成订单写入数据库 -> 卖家后台查收 -> 卖家调用 API 修改为“已出货” -> 买家确认收货 -> 交易闭环。

### 5.3 CI/CD 持续集成与持续交付流程 (核心亮点)
为了保障上述多阶段技术路线的高效推进，降低 4 人团队的代码合并冲突与部署风险，本项目全面引入 CI/CD 自动化流水线（依托 GitHub Actions 或 GitLab CI）：

1. **持续集成 (Continuous Integration - CI)**：
   * **代码提交与规范化检查**：任何团队成员向代码仓库 `push` 或提交 `Pull Request` 时，流水线自动触发代码风格检查。
   * **自动化测试**：自动化运行后端接口的单元测试（Pytest）和前端组件测试（Jest），如果测试覆盖率不达标或用例失败，将阻止代码合并，确保主干分支的绝对纯洁。
2. **持续交付/部署 (Continuous Deployment - CD)**：
   * **自动化构建**：代码合并至主分支后，CI/CD 工具会自动将前后端代码打包，并构建为轻量级的 Docker 镜像。
   * **自动化部署**：通过写好的部署脚本，将 Docker 镜像一键推送到测试服务器或生产服务器进行部署。前端静态资源（Vue）自动更新到 Nginx 目录，后端（Django）自动执行数据库迁移并重启服务。
3. **效益分析**：彻底告别“手动打包、手动传服务器”的刀耕火种时代。开发人员只需专注于写代码和提交，极大地节省了部署时间，保证了多端架构联调的高效性。

### 5.4 影响(或要求)
* **设备**：开发人员需配备普通 PC（推荐至少 8GB 内存以运行前端编译及 Docker），CI/CD 自动化依赖于代码托管平台的云端服务器。
* **软件**：操作系统（Windows/Mac/Linux），后端（Python/Django），前端（Node.js/Vue/Flutter），环境隔离（Docker），数据库（MySQL/PostgreSQL）。
* **经费**：所用核心技术及 CI/CD 工具（如 GitHub 免费额度）均为开源或免费，经费预估为 0。

## 6 经济可行性(成本----效益分析)

### 6.1 投资
本项目为教学实践项目，不涉及实际基本建设投资（硬件采购与人员工资薪酬），主要投资为团队 4 名成员的技术学习时间、编码精力和协作沟通成本。

### 6.2 预期的经济效益
虽无直接商业现金收益，但其“收益”具有长远价值：
1. 团队成员掌握了前后端分离、Flutter 跨平台开发以及现代 CI/CD 工业级流水线部署。
2. 高质量的项目产出带来了高学分收益以及丰富的求职简历素材。
整体来看，技能收益与教学收益远大于时间成本投入，经济上完全可行。

## 7 技术可行性(技术风险评价)

本团队现有 4 名成员合理分工（产品兼测试、前端开发、后端开发、DevOps 兼部署）。
* **架构风险低**：Django 框架封装度高，配合 Django REST Framework 能极快产出稳定 API，降低了后端开发难度。
* **演进路线平滑**：从 Vue 到 uni-app 的过渡几乎零学习成本，即便后期引入 Flutter 存在一定学习曲线，但由于前后端已经彻底分离，不影响现有系统的稳定运行。
* **工程质量保障**：CI/CD 机制的引入，极大降低了“代码在我的电脑上能跑，在服务器上报错”的环境不一致风险，确保了敏捷迭代的稳定性。
综合评估，具备充分的技术可行性。
