# 软件架构描述（SAD）

## 1 引言

### 1.1 目的与范围

本文档为商品网上交易系统（TradeX）的软件架构描述，旨在从多个架构视点系统性地阐述系统的逻辑结构、数据组织与接口设计。文档面向项目全体利益相关者，为系统工程师、开发人员、测试工程师及最终用户提供统一的架构参照基线，确保各方在系统设计意图上达成一致理解。

本文档的范围覆盖 TradeX 系统从后端业务服务、前端多终端应用到基础通信层的完整技术架构。后端采用 Python Django 框架构建 RESTful API 服务与 WebSocket 实时通信服务，前端涵盖 Vue 3 Web 端、Flutter 移动端与 uni-app 小程序端三个终端形态。文档不涉及具体算法实现细节或逐行代码说明，而是聚焦于架构层面的模块划分、层次依赖、数据流转与接口契约。

### 1.4 架构描述概览

本文档按照 IEEE 42010 的核心概念组织。第 2 章识别系统的利益相关者及其核心关注点，建立架构描述的需求基础。第 3 章定义三个架构视点——逻辑功能视点、数据视点与接口视点，分别阐明各视点所覆盖的关注点、采用的模型类型及其典型涉众。第 4 章在各视点的指导下展开具体的架构视图，提供组件划分模型、层次依赖模型、全局数据模型与接口契约模型。第 6 章建立视图之间的对应关系并给出一致性检查结论。第 7 章记录架构关键决策的背景、备选方案及决策理由。附录包含术语表与图表索引。

## 2 利益相关者与关注点

### 2.1 利益相关者识别

本系统的利益相关者共分五类。系统工程师与架构师负责系统的整体技术方案设计，关注模块划分是否合理、层次依赖是否清晰、技术选型是否匹配项目约束。项目经理负责项目进度与资源协调，关注架构方案的可实施性以及架构文档是否为团队协作提供了充分的指导。开发组长负责具体模块的编码实现与代码质量管控，关注模块间接口契约是否明确、数据结构是否稳定、开发分工是否可独立并行。测试工程师负责验证系统功能与非功能需求的满足度，关注接口的输入输出规格是否可测试、系统行为是否可预测。最终用户与需方即课程教学单位与普通网络购物者，关注系统是否满足业务需求、操作是否流畅、多端体验是否一致。

### 2.2 核心关注点列表

**FC-01 功能分解与模块低耦合**。系统的业务功能覆盖用户管理、商品管理、审查管理、订单交易、推荐系统、聊天系统、论坛社区等多个领域。架构需将这些功能合理分解为独立模块，模块间通过明确定义的接口交互，实现高内聚低耦合，使单个模块的修改不波及其他模块。

**FC-02 全局数据一致性与存储策略**。系统的核心数据包括用户信息、商品信息、订单交易数据、聊天记录等，这些数据在多个模块间共享。架构需确保数据的一致性与完整性，明确数据的存储策略、访问路径与事务边界，特别是在订单创建与库存扣减等关键场景中保障原子性。

**FC-03 运行期并发与实时响应**。系统需同时处理 HTTP 请求与 WebSocket 长连接两类流量。HTTP 请求面向商品检索、订单操作等业务场景，WebSocket 长连接面向实时聊天场景。架构需在运行期合理调度并发资源，确保两类流量互不干扰且均满足响应时间要求。

**FC-04 内外部接口标准化与兼容性**。系统对外面向 Web 端、Flutter 移动端、uni-app 小程序端三类客户端，对内集成第三方 AI 服务与腾讯云对象存储服务。架构需定义统一的接口协议与数据格式标准，确保多终端兼容性与第三方服务的可替换性。

**FC-06 资源利用效率与硬件约束**。作为教学实践项目，系统的服务器资源有限，推荐配置为 4 核 CPU、8GB 内存。架构需在有限资源下保障系统运行效率，数据库采用轻量级 SQLite，部署采用双进程（HTTP + WebSocket）模式，避免引入重型中间件。

## 3 架构视点定义

### 3.1 VP-01 逻辑功能视点

逻辑功能视点覆盖关注点 FC-01（功能分解与模块低耦合），旨在呈现系统的静态功能结构。该视点采用的模型类型包括组件图（描述后端 API 模块与前端应用的划分）、包图（描述模块内部的代码组织）以及层次结构图（描述从接入层到数据层的垂直调用关系）。该视点的典型涉众为开发组长与系统工程师，前者依据组件划分进行任务分配与并行开发，后者据此评估架构的合理性与可演进性。

### 3.2 VP-02 数据视点

数据视点覆盖关注点 FC-02（全局数据一致性与存储策略），旨在呈现系统的数据组织方式与流转规则。该视点采用的模型类型包括 ER 图（描述实体间的关联关系）、数据字典（描述每个数据实体的字段定义与约束）以及数据流图（描述关键业务场景中数据的生产、消费与变换路径）。该视点的典型涉众为数据库工程师与开发组长，前者据此完成物理数据库的建表与索引优化，后者据此理解业务数据在模块间的流转方式。

### 3.3 VP-05 接口视点

接口视点覆盖关注点 FC-04（内外部接口标准化与兼容性），旨在呈现系统的对外接口契约与通信机制。该视点采用的模型类型包括接口定义表（列举全部 API 端点的路径、方法、参数与响应格式）、协议栈图（描述 HTTP/HTTPS 与 WebSocket 两条通信链路的协议层次）以及数据元素规格表（定义接口传输的核心数据元素的类型、长度与约束）。该视点的典型涉众为系统集成工程师与测试工程师，前者据此完成多端与后端的联调集成，后者据此设计接口自动化测试用例。

## 4 架构视图

### 4.1 View-01 逻辑架构视图

#### 4.1.1 Model-L1 组件划分模型

系统按照前后端分离原则划分为后端服务集群与前端应用集群两大部分，各部分再按照业务领域或终端形态进一步拆分为独立组件。

后端服务以 Django 项目 `tradeX` 为配置容器，核心业务逻辑集中在 `core` 应用中。`core` 应用的 API 层按照业务领域拆分为 15 个独立路由模块，每个模块对应一个 Python 文件，封装该领域的全部 HTTP 端点。认证模块（`auth.py`，标识符 MOD-AUTH）负责用户登录与 JWT 令牌签发。用户模块（`users.py`，标识符 MOD-USER）负责用户注册、个人信息管理、收货地址更新与用户标签偏好管理。商品模块（`products.py`，标识符 MOD-PRODUCT）负责商品的发布、编辑、搜索、状态变更与卖家统计。分类模块（`categories.py`，标识符 MOD-CATEGORY）负责商品分类的增删改查与启用禁用。标签模块（`tags.py`，标识符 MOD-TAG）负责商品标签与商品标签关联的管理。收藏模块（`favorites.py`，标识符 MOD-FAVORITE）负责用户对商品的收藏与取消收藏操作。推荐模块（`recommendations.py`，标识符 MOD-RECOMMEND）负责热门商品推荐、个性化推荐与相似商品推荐。订单模块（`orders.py`，标识符 MOD-ORDER）负责订单创建、支付、发货、收货、取消等全生命周期状态管理及订单日志记录。通知模块（`notifications.py`，标识符 MOD-NOTIFICATION）负责订单通知、系统通知与消息通知的推送与已读管理。聊天模块（`chat.py`，标识符 MOD-CHAT-REST）提供聊天消息的 REST API 查询能力，包括会话列表、历史消息与未读计数。文件上传模块（`uploads.py`，标识符 MOD-UPLOAD）负责商品图片上传至腾讯云 COS 对象存储。RAG 模块（`rag.py`，标识符 MOD-RAG）负责与第三方 AI 模型的对话交互与基于向量检索的商品推荐。论坛模块（`forum.py`，标识符 MOD-FORUM）负责帖子、评论、点赞与论坛分类标签的管理。后台管理模块（`admin.py`，标识符 MOD-ADMIN）负责用户注册审核、商品审核以及系统管理功能。系统模块（`system.py`，标识符 MOD-SYSTEM）提供健康检查端点。

在 API 层之下，WebSocket 通信层由 `consumers.py`（标识符 MOD-CHAT-WS）实现，该模块基于 Django Channels 的 `AsyncWebsocketConsumer` 构建，负责实时聊天的连接认证、消息转发、已读回执与输入状态同步。`routing.py` 将 WebSocket 路由 `/ws/chat/` 映射到该消费者。

数据层由 `models.py`（标识符 MOD-MODEL）统一定义，包含 User、Category、Product、RegisterReview、ProductReview、Order、OrderDetail、Message、Tag、ProductTag、UserTagPreference、ProductFavorite、ForumCategory、ForumTag、ForumPost、ForumPostTag、ForumPostLike、ForumComment、OrderLog、Notification、ChatMessage 共 21 个 ORM 模型类。所有模型通过 Django ORM 映射至 SQLite 数据库表，业务层不直接执行原生 SQL。

前端应用集群包含三个独立终端。Web 端（标识符 APP-WEB）基于 Vue 3 与 Vite 构建，部署目录为 `webview/`，开发时通过 Vite 代理将 `/api/*` 请求转发至后端。Flutter 移动端（标识符 APP-MOBILE）部署目录为 `mobile/`，直接调用后端 HTTP 与 WebSocket 接口。uni-app 小程序端（标识符 APP-MINI）部署目录为 `mini-program/`，编译为微信小程序发布。三个终端共享同一套后端 API，接口协议与数据格式完全一致。

#### 4.1.2 Model-L2 层次依赖模型

系统采用四层架构，各层之间遵循单向依赖原则，上层依赖下层，禁止下层反向调用上层。

最上层为接入层。该层包含 Web 端、Flutter 移动端与 uni-app 小程序端三个前端应用。接入层通过 HTTP/HTTPS 协议调用业务层暴露的 RESTful API 端点，通过 WebSocket 协议连接业务层的实时聊天服务。接入层不直接访问数据层，所有数据请求均经由业务层中转。

第二层为业务层。该层是系统的核心处理层，由 django-ninja 框架驱动的 15 个 REST API 路由模块与 Django Channels 驱动的 WebSocket 消费者共同构成。每个路由模块接收接入层的请求，执行参数校验、身份认证与权限校验后，调用数据层的 ORM 模型完成数据读写操作，最终将处理结果序列化为 JSON 格式返回给接入层。业务层还负责调用外部服务：RAG 模块通过 OpenRouter API 调用第三方大语言模型，文件上传模块通过 COS SDK 调用腾讯云对象存储服务。业务层内部各模块之间通过 Django ORM 的模型关联关系间接交互，不存在模块间的直接函数调用依赖。

第三层为数据层。该层由 Django ORM 与 SQLite 数据库引擎共同构成。ORM 层将业务层的对象操作自动转换为标准 SQL 语句执行，并将查询结果映射回 Python 模型对象。SQLite 数据库以单文件形式存储全部业务数据（`backend/db.sqlite3`），支持完整的 ACID 事务特性。向量数据库 ChromaDB 作为数据层的补充组件，存储商品的向量化表示以支持 RAG 模块的语义检索能力。

最底层为基础设施层。该层提供运行环境支撑，包括操作系统（Linux/macOS）、Python 运行时（3.12）、Daphne ASGI 服务器（同时承载 HTTP 与 WebSocket）、Nginx 反向代理（生产环境）、Docker 容器化部署以及 GitHub Actions CI/CD 自动化流水线。基础设施层对上层透明，业务代码不直接依赖基础设施层的具体实现。

层次间的调用规则如下。接入层仅可调用业务层的公开 API 端点与 WebSocket 端点。业务层仅可调用数据层的 ORM 接口与外部服务的 SDK 接口。数据层仅可调用基础设施层提供的数据库引擎与文件系统接口。任何跨层调用或反向调用均为架构违规。

### 4.2 View-02 数据架构视图

#### 4.2.1 Model-D1 全局常量模型

系统定义了多组业务枚举常量，作为数据层与业务层共同遵守的值域约束。

用户角色枚举（`User.RoleChoices`）定义两个取值：`NORMAL`（普通用户）与 `ADMIN`（系统管理员），该枚举决定用户可访问的功能范围与后台管理权限。

用户注册状态枚举（`User.RegisterStatusChoices`）定义三个取值：`PENDING`（待审核）、`APPROVED`（审核通过）与 `REJECTED`（审核驳回）。新注册用户初始状态为 `PENDING`，需管理员审核后方可正常使用系统。

商品状态枚举（`Product.StatusChoices`）定义四个取值：`PENDING`（待审核）、`APPROVED`（审核通过）、`OFF_SHELF`（已下架）与 `REJECTED`（审核驳回）。仅状态为 `APPROVED` 的商品可在前端展示与被下单购买。

订单状态枚举（`Order.StatusChoices`）定义五个取值：`PENDING_PAY`（待付款）、`PENDING_SHIP`（已付款待出货）、`SHIPPED`（已出货待收货）、`COMPLETED`（已收货交易完成）与 `CANCELED`（已取消）。订单状态的合法转换路径由 `Order.can_transition_to()` 方法严格控制，待付款可转为已付款待出货或已取消，已付款待出货可转为已出货或已取消，已出货可转为已完成或已取消，已完成与已取消为终态不可再转换。

审查结果枚举（`RegisterReview.ResultChoices` 与 `ProductReview.ResultChoices`）均定义两个取值：`APPROVED`（审核通过）与 `REJECTED`（审核驳回）。

订单操作类型枚举（`OrderLog.ActionChoices`）定义六个取值：`CREATE`（创建订单）、`PAY`（支付订单）、`SHIP`（卖家发货）、`RECEIVE`（确认收货）、`CANCEL`（取消订单）与 `AUTO_COMPLETE`（自动完成）。

通知类型枚举（`Notification.TypeChoices`）定义三个取值：`ORDER`（订单通知）、`SYSTEM`（系统通知）与 `MESSAGE`（消息通知）。

论坛帖子状态枚举（`ForumPost.StatusChoices`）定义两个取值：`PUBLISHED`（已发布）与 `HIDDEN`（已隐藏）。

上述全部枚举常量均在 `backend/core/models.py` 中以 Django `TextChoices` 类型定义，由数据库字段的 `choices` 参数实施约束，业务层无需额外校验枚举合法性。

#### 4.2.2 Model-D2 全局变量模型

系统的全局运行时变量由 Django 配置文件 `backend/tradeX/settings.py` 统一管理，通过环境变量注入实现多环境适配。

`ENVIRONMENT` 变量决定系统加载的配置集，取值为 `local`（本地开发）、`development`（远程开发）或 `production`（生产环境），分别对应 `.env.local`、`.env.development`、`.env.production` 三个环境配置文件。

`SECRET_KEY` 变量用于 JWT 令牌签名与 Django 内部加密操作，各环境使用不同的密钥值，生产环境必须使用高强度随机密钥。`DEBUG` 变量控制调试模式开关，本地与远程开发环境为 `True`，生产环境为 `False`。`ALLOWED_HOSTS` 变量限定可接受的请求主机名列表。`DATABASE_URL` 变量指定数据库连接地址，默认为 SQLite 文件路径 `sqlite:///db.sqlite3`，亦支持 PostgreSQL 连接串。

`OPENROUTER_API_KEY` 与 `OPENROUTER_BASE_URL` 两个变量配置第三方 AI 服务的认证密钥与接口地址，由 RAG 模块在运行时读取。腾讯云 COS 相关变量配置对象存储的密钥、存储桶与区域信息，由文件上传模块读取。

`deploy-config.json` 文件作为前端与部署脚本共享的全局配置，记录当前环境下后端 HTTP 服务与 WebSocket 服务的地址。本地开发环境下 HTTP 地址为 `http://localhost:7001`，WebSocket 地址为 `ws://localhost:7002/ws/chat/`。通过 `script/switch-config.sh` 脚本切换环境时自动更新该文件的内容。

WebSocket 消费者 `ChatConsumer` 维护一个类级别的 `user_connections` 字典，以用户 ID 为键记录当前在线连接数，用于实施单用户最多 3 个并发连接的限制策略。该变量随进程生命周期存在，进程重启后归零。

#### 4.2.3 Model-D3 数据结构模型

系统的持久化数据结构由 21 个 ORM 模型组成，按业务领域划分为六个实体组。

**用户与权限实体组**包含 User 一个实体。User 实体以 `user_id`（字符串，最长50字符）为主键，存储用户名、邮箱、电话、加密密码、真实姓名、身份证号、联系电话、收货地址、角色、注册状态、注册时间与最后登录时间。用户名、邮箱、电话与身份证号均设有唯一约束。密码字段采用 Django 原生 PBKDF2 哈希算法加密存储。

**商品与分类实体组**包含 Category、Product、Tag、ProductTag 四个实体。Category 实体以 `category_id` 为主键，存储分类名称、描述、排序序号与启用状态，支持软删除（`is_active` 置为 `False`）。Product 实体以 `product_id` 为主键，通过 `publisher` 外键关联 User，通过 `category_ref` 外键关联 Category（`SET_NULL` 策略），存储商品名称、描述、图片地址、价格、库存、浏览量、销量、收藏数、平均评分、商品状态、发布时间与审核时间。Tag 实体以 `tag_id` 为主键，存储标签名称、标签分类与使用次数。ProductTag 实体作为 Product 与 Tag 的多对多关联表，存储标签权重与打标时间，以 `(product, tag)` 联合唯一约束防止重复关联。

**审查实体组**包含 RegisterReview 与 ProductReview 两个实体。RegisterReview 以 `review_id` 为主键，通过 `pending_user` 一对一关联 User（被审查用户），通过 `admin` 外键关联 User（审查管理员），存储审查结果、审查意见与审查时间。ProductReview 结构类似，`pending_product` 一对一关联 Product。

**订单与交易实体组**包含 Order、OrderDetail 与 OrderLog 三个实体。Order 以 `order_id` 为主键，通过 `buyer` 与 `seller` 两个外键分别关联 User（`PROTECT` 策略，防止用户删除导致订单丢失），存储订单总金额、收货地址快照、联系电话快照、订单状态、下单时间、付款时间、发货时间、收货时间、自动确认收货时间、物流公司、物流单号、物流信息快照（JSON 字段）与取消原因。地址与电话采用快照策略，下单时从用户当前信息复制一份，后续用户修改地址不影响已有订单。OrderDetail 以 `detail_id` 为主键，通过外键关联 Order 与 Product，存储购买数量、下单单价快照与小计金额。OrderLog 以 `log_id` 为主键，通过外键关联 Order 与操作人 User，存储操作类型、原状态、新状态、备注与操作时间，实现订单全生命周期操作的不可篡改审计追踪。

**社交与互动实体组**包含 Message、ProductFavorite、UserTagPreference、ChatMessage、Notification 五个实体。Message 实体存储商品留言与商家回复。ProductFavorite 以 `(user, product)` 联合唯一约束实现收藏关系。UserTagPreference 以 `(user, tag)` 联合唯一约束存储用户对标签的偏好分数，供推荐系统计算个性化推荐。ChatMessage 存储用户间的聊天消息，通过 `sender` 与 `receiver` 外键关联 User，可选关联 Product 与 Order 以建立聊天与业务实体的上下文关系，并建立了 `(sender, receiver)`、`(receiver, is_read)` 与 `created_at` 三个索引以加速会话查询与未读计数。Notification 存储面向用户的系统通知，可选关联 Order。

**论坛实体组**包含 ForumCategory、ForumTag、ForumPost、ForumPostTag、ForumPostLike、ForumComment 六个实体。ForumPost 通过 `author` 外键关联 User，通过 `category` 外键关联 ForumCategory，存储帖子标题、内容、封面图片、浏览量、点赞数、评论数、状态、是否置顶、发布时间与更新时间。ForumPostTag 作为帖子与标签的多对多关联表。ForumPostLike 以 `(user, post)` 联合唯一约束实现点赞关系。ForumComment 通过外键关联 ForumPost 与 User，支持软删除。

实体间的核心关联关系遵循以下规则。User 与 Product 为一对多关系，一个用户可发布多个商品。User 与 Order 有两条一对多关系，分别代表买家维度与卖家维度。Category 与 Product 为一对多关系。Order 与 OrderDetail 为一对多关系。Product 与 Tag 通过 ProductTag 构成多对多关系。User 与 Tag 通过 UserTagPreference 构成多对多关系。User 与 RegisterReview 为一对一关系（被审查维度），User 与 RegisterReview 为一对多关系（审查管理员维度）。Product 与 ProductReview 同理。全部外键的删除策略根据业务语义选择 CASCADE（级联删除，用于强归属关系）、SET_NULL（置空，用于可选关联）或 PROTECT（禁止删除，用于核心交易关系）。

### 4.3 View-05 接口架构视图

#### 4.3.1 Model-I1 接口标识与拓扑模型

系统的外部接口分为两条独立的通信链路。第一条为 HTTP/HTTPS 链路，承载全部 RESTful API 请求，后端服务默认监听 `0.0.0.0:8000`（开发环境映射为 7001 端口）。第二条为 WebSocket 链路，承载实时聊天消息，后端服务默认监听 `0.0.0.0:8001`（开发环境映射为 7002 端口）。两条链路由同一个 Daphne ASGI 服务器的两个进程分别承载，共享同一个 Django 项目配置与数据库实例。

HTTP 链路上，所有 API 端点挂载在 `/api/` 前缀之下，由 django-ninja 框架自动生成 OpenAPI 规范文档，可通过 `/api/docs` 访问 Swagger UI。WebSocket 链路上，聊天端点路径为 `/ws/chat/`，通过查询参数 `token` 传递 JWT 令牌进行连接认证。

按业务领域，HTTP API 端点的路由映射如下。认证相关端点挂载于 `/api/login`。后台管理端点挂载于 `/api/admin`。用户相关端点挂载于 `/api/users`。商品相关端点挂载于 `/api/products`。分类相关端点挂载于 `/api/categories`。标签相关端点挂载于 `/api/tags`。商品标签关联端点挂载于 `/api/product-tags`。用户标签偏好端点挂载于 `/api/user-tag-preferences`。商品收藏端点挂载于 `/api/product-favorites`。论坛端点挂载于 `/api/forum`。推荐系统端点挂载于 `/api/recommendations`。订单端点挂载于 `/api/orders`。通知端点挂载于 `/api/notifications`。文件上传端点挂载于 `/api/uploads`。RAG AI 端点挂载于 `/api/rag`。健康检查端点挂载于 `/api/health`。聊天 REST 端点挂载于 `/api/chat`。

外部第三方服务接口包括两个。OpenRouter AI 接口（IF-EXT-AI）为 HTTPS 协议出站调用，由 RAG 模块通过 OpenAI Python SDK 发起，目标地址由 `OPENROUTER_BASE_URL` 配置。腾讯云 COS 接口（IF-EXT-COS）为 HTTPS 协议出站调用，由文件上传模块通过 COS Python SDK 发起。

#### 4.3.2 Model-I2 数据元素契约模型

系统的 API 接口请求与响应均采用 JSON 格式编码。请求体的 `Content-Type` 头部为 `application/json`（文件上传端点除外，采用 `multipart/form-data`）。响应体统一包裹在 JSON 对象中，成功响应直接返回业务数据，失败响应返回包含错误码与错误描述的对象。

核心数据元素的契约定义如下。`user_id` 为字符串类型，最大长度 50 字符，格式为 UUID 或自定义前缀加随机串（如 `seed_admin`），在用户注册时由服务端生成，全局唯一。`product_id` 为字符串类型，最大长度 50 字符，在商品发布时由服务端生成。`order_id` 为字符串类型，最大长度 50 字符，在订单创建时由服务端生成。`price` 与 `total_amount` 为十进制数类型，精确到小数点后两位，分别最大 10 位与 12 位有效数字。`stock` 与 `quantity` 为非负整数类型。`image_url` 为有效 URL 字符串，最大长度 500 字符。时间类字段统一采用 ISO 8601 格式字符串（如 `2026-06-11T10:30:00Z`）。

身份认证令牌采用 JWT 格式。令牌载荷（payload）包含 `user_id` 字段标识令牌持有者，`exp` 字段标识令牌过期时间。令牌通过 HMAC-SHA256 算法签名，签名密钥为 Django `SECRET_KEY`。客户端在 HTTP 请求中通过 `Authorization: Bearer <token>` 头部传递令牌，在 WebSocket 连接中通过查询参数 `token=<token>` 传递令牌。

#### 4.3.3 Model-I3 数据集合体契约模型

系统的列表查询接口统一支持分页参数。`limit` 参数指定每页返回记录数，默认值为 10。`offset` 参数指定从第几条记录开始返回，默认值为 0。分页响应包含记录列表与总记录数，客户端据此计算总页数与当前页位置。

商品搜索接口 `GET /api/products/search/` 支持组合查询：`q` 参数为关键词模糊搜索（匹配商品名称与描述），`category` 参数为分类 ID 精确筛选，`sort` 参数指定排序方式（如 `time` 按发布时间倒序）。当 `q` 为空且 `category` 有值时，返回该分类下全部商品；当两者均有值时，在指定分类内执行关键词搜索。

推荐系统接口根据用户登录状态返回不同的集合。已登录用户调用 `GET /api/recommendations/personalized/` 获取基于标签偏好的个性化推荐，未登录用户调用 `GET /api/recommendations/trending/` 获取基于热度排序的热门推荐。两个接口均支持 `limit` 与 `offset` 分页参数。

聊天系统 REST 接口返回的会话列表按最新消息时间倒序排列，每条会话记录包含对方用户信息、最新一条消息内容、未读消息计数。历史消息接口按时间正序返回指定两个用户之间的聊天记录。

#### 4.3.4 Model-I4 通信方法模型

HTTP 通信采用标准的 RESTful 方法语义。GET 方法用于查询操作，不产生副作用。POST 方法用于创建操作，如注册用户、发布商品、创建订单。PUT 方法用于全量更新操作，如更新商品信息、更新分类信息。PATCH 方法用于部分更新操作，如修改商品状态、修改订单状态。DELETE 方法用于删除操作（部分场景为逻辑删除）。

WebSocket 通信采用基于消息类型的事件分发模式。客户端发送的消息必须为 JSON 格式，包含 `type` 字段标识消息类型。`chat_message` 类型表示聊天消息，须附带 `to`（接收者用户 ID）与 `content`（消息内容）字段，可选附带 `product_id` 与 `order_id` 字段建立业务上下文。`ping` 类型表示心跳检测，服务端收到后立即回复 `pong` 类型消息。`read_receipt` 类型表示已读回执，须附带 `message_id` 字段。`typing` 类型表示正在输入状态，须附带 `to` 与 `typing`（布尔值）字段。

服务端向客户端推送的消息类型包括 `chat_message`（转发的聊天消息，附带 `message_id`、`from`、`content`、`created_at` 等字段）、`message_sent`（发送成功回执）、`read_receipt`（对方已读回执）、`typing`（对方输入状态）、`system`（系统消息）与 `error`（错误消息）。

#### 4.3.5 Model-I5 协议特性模型

HTTP 链路的协议栈自底向上为 TCP → TLS（生产环境）→ HTTP/1.1。开发环境下省略 TLS 层，直接使用 HTTP 明文传输。生产环境下由 Nginx 反向代理终止 TLS，后端 Daphne 服务接收 Nginx 转发的明文 HTTP 请求。请求超时时间由客户端自行控制，服务端不设强制超时，但建议客户端对普通接口设置 10 秒超时、对 RAG AI 接口设置 60 秒超时（因大语言模型推理耗时较长）。

WebSocket 链路的协议栈自底向上为 TCP → TLS（生产环境）→ HTTP/1.1 Upgrade → WebSocket。客户端首先通过 HTTP GET 请求发起协议升级握手，服务端验证 JWT 令牌后接受升级，后续通信转为 WebSocket 全双工模式。心跳机制由客户端主动发送 `ping` 消息实现，建议间隔 30 秒发送一次，服务端记录最近心跳时间用于连接活性检测。单用户并发连接数上限为 3，超出限制的新连接将被拒绝（关闭码 4005）。连接认证失败的关闭码定义如下：4001 表示缺少 token 参数，4002 表示 token 中无 user_id，4003 表示 token 已过期，4004 表示 token 无效。

#### 4.3.6 Model-I6 物理兼容性模型

系统的三类前端终端在物理层面存在差异化的兼容性需求。Web 端运行在 PC 浏览器环境中，要求 Chrome 90+、Edge 90+ 或 Firefox 88+ 版本，通过浏览器原生的 `fetch` API 与 `WebSocket` API 访问后端服务。Flutter 移动端运行在 Android 10+ 与 iOS 14+ 操作系统上，通过 Dart 的 `http` 包与 `web_socket_channel` 包访问后端服务。uni-app 小程序端运行在微信客户端中，通过微信小程序原生的 `wx.request` 与 `wx.connectSocket` API 访问后端服务。

三类终端对同一套后端 API 的调用方式存在细微差异。Web 端在开发环境下通过 Vite 的 `proxy` 配置将 `/api/*` 请求代理至后端，避免跨域问题；生产环境下通过 Nginx 同源代理实现。Flutter 移动端与 uni-app 小程序端直接向后端地址发起跨域请求，后端通过 `django-cors-headers` 中间件配置允许的跨域源。WebSocket 连接地址需根据环境从 `deploy-config.json` 中读取，确保各终端连接正确的服务实例。

## 6 对应关系与一致性

### 6.1 对应规则定义

本架构描述中三个视图之间存在以下对应规则。

规则 CR-01：逻辑架构视图中的每个 API 模块组件（Model-L1）必须在接口架构视图（Model-I1）中有对应的路由映射条目。具体而言，MOD-AUTH 对应 `/api/login`，MOD-USER 对应 `/api/users`，MOD-PRODUCT 对应 `/api/products`，以此类推，共 17 个模块与 17 组路由的一一映射。

规则 CR-02：数据架构视图中的每个 ORM 模型实体（Model-D3）必须在逻辑架构视图的数据层（Model-L2）中有对应的存储映射，且至少被一个业务层模块通过 ORM 接口访问。

规则 CR-03：接口架构视图中每个数据元素（Model-I2）的类型与约束必须与数据架构视图中对应模型字段（Model-D3）的定义一致。例如，接口中 `price` 的十进制两位小数约束必须与 Product 模型的 `DecimalField(max_digits=10, decimal_places=2)` 定义吻合。

规则 CR-04：全局常量模型（Model-D1）中定义的枚举取值范围必须与接口契约中该字段的合法取值范围一致。例如，订单状态的五个枚举值必须同时约束数据库层的写入校验与接口层的参数校验。

### 6.2 一致性检查结果

针对上述四条对应规则逐一检查，结论如下。

CR-01 检查通过。`core/api/__init__.py` 中的路由注册代码通过 `router.add_router()` 将 15 个业务模块与 2 个辅助模块（商品标签、用户标签偏好）分别挂载到唯一的 URL 前缀下，`routing.py` 将 WebSocket 消费者挂载到 `/ws/chat/` 路径，全部模块均有对应路由，无遗漏。

CR-02 检查通过。21 个 ORM 模型均定义在 `core/models.py` 中，由 Django 迁移机制自动映射为 SQLite 数据库表。每个模型至少被一个 API 模块的视图函数通过 `objects.filter()`、`objects.create()` 等 ORM 接口访问。

CR-03 检查通过。接口层通过 django-ninja 的 Schema 类定义请求与响应的数据类型，Schema 类的字段类型与 ORM 模型的字段类型保持严格对应。django-ninja 在序列化时自动将 `DecimalField` 转换为 JSON 数字，将 `DateTimeField` 转换为 ISO 8601 字符串，类型转换逻辑一致。

CR-04 检查通过。所有枚举字段在 ORM 模型中通过 `choices` 参数约束，在 API 层通过 Schema 的枚举类型或业务逻辑校验约束，两层约束的取值范围完全一致。

## 7 架构决策记录

### 7.1 ADR-001 模块划分策略

**背景**：系统的业务功能涵盖用户管理、商品管理、订单交易、聊天、论坛、推荐、AI 等多个领域，开发团队为 4 人小组，需要支持并行开发且降低模块间的代码冲突概率。

**备选方案一**：单文件 API 方案，将全部 API 端点定义在一个 `api.py` 文件中。该方案实现简单，但随着功能增长文件将膨胀至数千行，多人协作时冲突频繁。

**备选方案二**：按业务领域拆分多文件方案，每个业务领域对应一个独立的 Python 文件与独立的 Router 实例，由 `__init__.py` 聚合挂载。该方案模块边界清晰，多人可并行开发不同模块而互不干扰。

**决策**：采用方案二。系统早期版本确实采用过方案一的单文件结构，随着聊天、论坛、订单等功能的加入，在 v0.2.1 版本中重构为多文件方案。重构后 `core/api/` 目录下包含 `auth.py`、`users.py`、`products.py` 等 15 个业务模块文件与 1 个公共定义文件 `common.py`，每个文件行数控制在 200-400 行之间，职责单一，便于阅读与维护。

### 7.2 ADR-002 全局数据管理策略

**背景**：系统的核心数据需要在多个模块间共享（如 User 数据被认证、订单、聊天等多个模块读取），需要选择合适的数据访问与一致性保障策略。

**备选方案一**：每个模块维护独立的数据存储，模块间通过事件或消息传递数据。该方案模块间解耦度最高，但在 4 人团队与 SQLite 数据库的约束下引入了不必要的复杂度。

**备选方案二**：全部模块共享同一个关系型数据库实例，通过 Django ORM 的模型层作为唯一的数据访问通道，利用数据库事务保障关键操作的原子性。该方案实现简洁，与 Django 生态天然契合。

**决策**：采用方案二。全部 21 个 ORM 模型定义在同一个 `models.py` 文件中，各业务模块通过导入所需的模型类直接访问数据库。订单创建场景需要同时写入 Order、OrderDetail 并扣减 Product 库存，这些操作被包裹在 Django 的 `transaction.atomic()` 上下文中以保障原子性。向量数据库 ChromaDB 作为补充组件，仅供 RAG 模块使用，不参与核心业务的数据一致性保障。

### 7.3 ADR-003 接口通信机制选型

**背景**：系统需要同时支持传统的请求-响应式业务操作与实时的聊天消息推送，需要选择合适的通信协议组合。

**备选方案一**：全部采用 HTTP 轮询方式，聊天消息通过客户端定时拉取实现。该方案实现最简单，但轮询间隔与消息实时性之间存在矛盾，且高频轮询会造成无效的服务器资源消耗。

**备选方案二**：全部采用 WebSocket 长连接方式，包括业务操作也通过 WebSocket 消息实现。该方案实时性最佳，但将常规 CRUD 操作包装为 WebSocket 消息会丧失 HTTP 语义的天然优势（如 RESTful 方法语义、状态码、缓存机制），且增加了客户端开发的复杂度。

**备选方案三**：混合方案，业务操作采用 HTTP RESTful API，实时聊天采用 WebSocket 长连接。两条链路独立运行，共享同一套身份认证机制（JWT 令牌）。该方案兼顾了业务操作的标准化与聊天消息的实时性。

**决策**：采用方案三。后端通过 Django Channels 框架在 ASGI 层同时承载 HTTP 与 WebSocket 两种协议，开发环境下由 `run-backend.sh` 脚本分别在两个端口启动两个 Daphne 进程。django-ninja 框架负责 HTTP 链路的路由分发与 OpenAPI 文档生成，Django Channels 的 `AsyncWebsocketConsumer` 负责 WebSocket 链路的连接管理与消息转发。两条链路共享同一个 Django 项目配置与数据库实例，聊天消息同时通过 WebSocket 实时推送与 REST API 历史查询两种方式提供访问。

### A 术语表

| 术语 | 定义 |
|------|------|
| 前后端分离 | 一种架构模式，前端负责界面渲染与用户交互，后端负责业务逻辑与数据处理，双方通过 API 接口通信 |
| django-ninja | 基于 Django 的高性能 API 框架，支持 Python 类型提示与自动 OpenAPI 文档生成 |
| Django Channels | Django 的异步扩展库，使 Django 支持 WebSocket、HTTP2 等非 HTTP 协议 |
| Daphne | Django Channels 官方推荐的 ASGI 服务器，支持 HTTP 与 WebSocket 协议 |
| ChromaDB | 开源的向量数据库，用于存储与检索文本的向量化表示 |
| OpenRouter | AI 模型路由服务，提供统一接口调用多种大语言模型 |
| 腾讯云 COS | 腾讯云对象存储服务，用于存储商品图片等静态资源 |
| 软删除 | 通过修改记录的状态标识（如 is_active 置为 False）实现逻辑删除，而非物理删除数据库记录 |
| 快照策略 | 在创建关联记录时复制关键字段的当前值，使后续源数据变更不影响已创建的记录 |
