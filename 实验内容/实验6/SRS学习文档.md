# 文档11《软件需求规格说明(SRS)》学习笔记

> 基于 GB/T 8567-2006《计算机软件文档编制规范》，梳理 SRS 文档的要求、结构及其与其他相关文档的关系。

---

## 一、文档定位

**文档编号**：11  
**文档全称**：软件需求规格说明（Software Requirements Specification, SRS）  
**所属标准**：GB/T 8567-2006《计算机软件文档编制规范》  
**文档角色**：**核心文档**——描述对软件产品的需求，是后续设计、编码、测试的基础依据

SRS 是软件开发生命周期中需求阶段的最终产出物，也是开发方与需方之间的"合同"。所有后续工作（设计、实现、测试）都需要回溯到 SRS 进行验证。

---

## 二、SRS 文档结构与要求

根据 GB/T 8567-2006，文档11 SRS 的标准章节结构如下：

### 1 引言

| 小节 | 内容要求 |
|------|----------|
| **1.1 标识** | 文档编号、标题、版本号、发行号，以及该文档所适用的软件系统标识 |
| **1.2 系统概述** | 系统用途、一般特性、开发与运行历史、项目相关方（投资方/需方/用户/开发方/支持机构等）、运行现场 |
| **1.3 文档概述** | 文档的用途、预期读者、保密性要求 |
| **1.4 基线** | 描述本规格说明编写的依据基线（如可行性分析报告、项目合同等） |

### 2 引用文件

列出本 SRS 引用的所有文档，包括编号、标题、版本、日期、来源。常见引用包括：
- GB/T 8567-2006 规范本身
- 可行性分析报告（FAR, 文档01）
- 运行概念说明（OCD, 文档06）
- 系统需求规格说明（SSS, 文档07）

### 3 需求

这是 SRS 的核心章节，描述软件的全部功能与非功能需求。

| 小节 | 内容要求 |
|------|----------|
| **3.1 要求的状态和方式** | 定义系统的运行状态（如未登录、已登录、管理员模式等），每个需求与此关联 |
| **3.2 需求概述** | 系统总体功能与业务结构，可包含功能结构图、业务流程图 |
| **3.3 需求规格** | 逐条详述功能需求，通常按模块组织（用户管理、商品管理等） |
| **3.4 非功能需求** | 性能、安全性、可用性、可维护性、接口等非功能约束 |
| **3.x CSCI 能力需求** | （可选）更细粒度的计算机软件配置项级别的需求 |

### 4 合格性规定

定义如何验证每条需求是否被满足（即需求的可测试性标准）。

### 5 需求可追踪性

| 小节 | 内容要求 |
|------|----------|
| **5.1 需求到系统设计的可追踪性** | 每条需求 → 对应哪个设计模块 |
| **5.2 需求到测试的可追踪性** | 每条需求 → 由哪个测试用例验证 |

### 6 注释

包含背景信息、词汇表、缩写词表，以及可选的附录。

---

## 三、SRS 核心编写原则

1. **完整性**：覆盖所有功能性需求和非功能性需求，不遗漏
2. **一致性**：需求之间不能自相矛盾；术语统一
3. **可验证性**：每条需求都应当能通过某种方法（测试、审查、演示）验证
4. **可追踪性**：每条需求有唯一标识，能向前追溯到来源、向后追溯到设计和测试
5. **无歧义性**：语言精确，避免模糊表述（如"尽可能快"应改为"响应时间≤200ms"）
6. **可行性**：需求在当前技术条件下可实现

## 四、与其他相关文档的关系

在 GB/T 8567-2006 规范体系中，SRS 不是孤立存在的。以下分析与文档07、08、12、17 的关系：

### 4.1 与文档07《系统需求规格说明(SSS)》的关系

| 维度 | 07-SSS | 11-SRS |
|------|--------|--------|
| **视角** | 系统级视角（含硬件、人员、软件） | 纯软件视角 |
| **受众** | 系统工程师、项目经理、需方高层 | 软件开发人员、测试人员 |
| **内容侧重** | 系统整体需求，包括硬件配置、网络拓扑、业务流程、操作流程 | 软件功能描述、接口、数据、性能等软件技术细节 |
| **粒度** | 较粗，面向系统分配 | 较细，可直接指导编码与测试 |

**关系**：SSS 是 SRS 的上游输入。SSS 将系统需求分解后，将软件部分的需求分配给 SRS 来详细描述。**SRS 必须与 SSS 保持需求一致性**，不能超出 SSS 的范围也不能遗漏 SSS 中分配给软件的需求。

**实际分工**：在 TradeX 项目中，SSS 描述"整个交易系统需要支持 Web + APP + 小程序多端访问"，而 SRS 进一步细化"后端 API 需要提供统一的 RESTful 接口，每个接口定义请求参数、返回格式、错误码"。

### 4.2 与文档08《接口需求说明(IRS)》的关系

**IRS（接口需求说明）** 是描述系统/软件之间接口需求的文档，内容包括：
- 接口的标识与名称
- 接口的输入/输出数据格式
- 接口的通信协议
- 接口的性能要求

**与 SRS 的关系**：
- **互补关系**：SRS 描述软件"做什么"，IRS 描述软件"怎么连接外部"。SRS 中的接口需求部分应当引用 IRS 中的详细定义，或在 SRS 中给出接口需求摘要，详细内容指向 IRS。
- **前后序关系**：IRS 通常在 SRS 之后或同步完成——先确定软件需要哪些功能（SRS），再确定这些功能依赖哪些接口（IRS）。
- **在小型项目中**，接口需求可以直接写入 SRS 的相应章节，而不单独成文。

**TradeX 实例**：
- SRS 描述："用户可通过图片上传功能为商品添加图片"
- IRS 进一步说明："图片上传接口 POST /api/uploads/image/，接收 multipart/form-data 格式，字段名 file，返回 JSON 含 url 和 key"

### 4.3 与文档12《数据需求说明(DRD)》的关系

**DRD（数据需求说明）** 描述软件对数据的需求，内容包括：
- 数据实体及其属性定义
- 数据的逻辑结构
- 数据的约束条件
- 数据的采集、存储、处理要求

**与 SRS 的关系**：
- **派生关系**：SRS 是 DRD 的上游。DRD 从 SRS 的功能需求中推导出数据需求——SRS 说"系统需要管理用户信息"，DRD 则定义"用户实体包含 user_id、username、email、phone 等字段"。
- **一致性约束**：DRD 中的数据定义必须满足 SRS 中描述的功能所需。如果 SRS 要求"用户可按分类浏览商品"，DRD 必须包含分类实体及其与商品的关联。
- **粒度差异**：SRS 关注"需要什么数据来支撑功能"，DRD 关注"数据本身的结构、类型、约束、关系"。

**TradeX 实例**：
- SRS："系统应能根据商品分类向用户推荐相关商品"
- DRD：定义 Category(分类) 实体 {category_id, name, description, sort_order}，Product(商品) 实体关联 category_id 外键

### 4.4 与文档17《软件配置管理计划(SCMP)》的关系

**SCMP（软件配置管理计划）** 描述如何对软件项目中的配置项（包括文档、代码、数据）进行标识、控制、状态记录和审计。内容包括：
- 配置项列表（哪些文件和文档受控）
- 版本控制策略
- 变更控制流程
- 基线管理
- 权限与审批机制

**与 SRS 的关系**：
- **管理关系**：SCMP 管理 SRS 本身。SRS 是 SCMP 中最核心的受控配置项之一。SCMP 定义：谁有权限修改 SRS？SRS 变更需要谁审批？SRS 的版本号如何命名？
- **流程支撑**：当需求变更时（文档 `qt-软件需求变更单`），SCMP 规定的变更流程决定 SRS 如何被修改——先提交变更申请 → 审批 → 修改 SRS → 更新基线。
- **工具支持**：SCMP 会规定使用什么工具（如 Git）来管理 SRS 的版本，以及分支策略。例如 TradeX 项目中，SRS 文件存放在 `GBT8567/系统需求规格说明-V1.0.md`，通过 Git 进行版本控制。

**TradeX 实例**：
- SCMP 规定：SRS 属于"技术文档"配置项，修改需至少一名评审人 approve，版本格式为 V<major>.<minor>
- 当 tradeX 项目新增论坛功能时：先提变更单 → 评审通过 → 修改 SRS（增加论坛需求章节）→ 版本从 V1.0 升至 V1.1

## 五、SRS 编写注意事项

1. **SRS 不是设计文档**：应描述"做什么"（What），而非"怎么做"（How）。例如写"用户登录需验证用户名和密码"，而非"使用 bcrypt 加密密码存储在 user 表"。
2. **与 SSS 的分工要清晰**：SSS 面向系统全局（含硬件部署、运维流程等），SRS 聚焦软件本身。TradeX 项目中 SSS 与 SRS 存在部分内容重叠，但视角不同。
3. **接口需求可引用 IRS**：SRS 中涉及外部接口的章节，应引用 IRS 文档编号，避免在 SRS 中重复写详细的接口定义。
4. **数据需求引用 DRD**：SRS 中的功能描述会涉及数据实体，参考 DRD 确保术语一致。
5. **版本受 SCMP 管控**：SRS 的所有修改必须遵循 SCMP 规定的变更流程，版本号在文档标识中明确。
